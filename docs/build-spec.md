# Install Build Spec — CORE Cowork Max Plugin

> **Status:** v1.0 — initial spec, authored 2026-05-11.
> **Owner:** Plugin repo (`dbatesai/core-claude-cowork-max-plugin`).
> **Consumers:** Any human or agent producing the Cowork-uploadable `.zip` artifact for this plugin.

This document is the authoritative spec for how the plugin source tree is packaged into a Cowork-installable `.zip` artifact. It defines inputs, outputs, version handling, the inclusion/exclusion contract, and the validation gates a build must pass before the artifact is considered releasable.

The actual build implementation (script or CI workflow) is a separate concern — but it MUST conform to this spec. If the script and the spec disagree, the spec wins and the script is wrong.

---

## §1. Purpose

Produce a single `.zip` artifact that:

1. **Installs cleanly** in Cowork via *Customize → Browse plugins → Add custom plugin*.
2. **Matches the plugin version** declared in `.claude-plugin/plugin.json`.
3. **Contains everything an installer needs** — manifests, hooks, scripts, MCP server, bundled skills, Live Artifacts, templates, connectors, end-user docs.
4. **Contains nothing that shouldn't ship** — no CORE-dev meta, no `.git/`, no Python build caches, no macOS noise, no in-progress notes, no release artifacts from prior builds.
5. **Is verifiable** — every release has a published SHA256 and survives the validation gates in §10.

## §2. Inputs

| Input | Source | Authority |
|---|---|---|
| Plugin source tree | `/` of this repo | Working tree at build time |
| Plugin version | `.claude-plugin/plugin.json` → `version` field | Sole source of truth — see §4 |
| Release notes | `CHANGELOG.md` — entry header `## [X.Y.Z] — YYYY-MM-DD` | Must match plugin.json version |
| Bundled skills | `skills/` subtree + `skills/CHECKSUMS.txt` | Regenerated by `scripts/refresh_bundled_skills.py` before release |

The build MUST NOT pull from the CORE dev project (`~/Documents/Projects/CORE/`). Anything needed at install time must already be present in this repo.

## §3. Output Artifact

| Property | Value |
|---|---|
| Filename | `core-claude-cowork-max-plugin-v{VERSION}.zip` |
| Default output location | `~/Desktop/` for local test builds; `dist/` (gitignored) for CI builds |
| Top-level directory inside zip | `core-claude-cowork-max-plugin/` |
| Compression | Standard `zip` deflate. No password. No encryption. |
| Maximum size | 50 MB (Cowork upload cap per [anthropics/claude-code#28337](https://github.com/anthropics/claude-code/issues/28337)). Build aborts above 45 MB to leave headroom. |
| Companion artifacts | `*.SHA256SUMS` (always); `*.asc` PGP signature (optional, for marketplace + SharePoint channels) |

The filename version segment MUST match `plugin.json` version exactly. No `-rc1`, `-beta`, `-test` suffixes in this build — those require their own spec amendment.

## §4. Versioning Model

**The plugin version lives in exactly one place: `.claude-plugin/plugin.json` → `version`.** Everything else derives from it.

### Coherence Rules

The build MUST verify all of the following before producing an artifact:

1. `plugin.json` version is valid SemVer (`X.Y.Z` with optional pre-release/build metadata per [semver.org](https://semver.org/spec/v2.0.0.html)).
2. `CHANGELOG.md` contains a section header `## [X.Y.Z] — YYYY-MM-DD` matching `plugin.json` version, with a date that is not in the future.
3. `CHANGELOG.md` does NOT contain an `## [Unreleased]` section — those entries must be promoted to a released version or removed before the build.
4. No prior release exists with the same version (a zip of that name should not be present in `dist/`).

If any of these fail, the build aborts with a clear error pointing at the file that needs to change.

### Bumping the Version

When David (or future maintainer) decides to ship a new release:

1. Edit `.claude-plugin/plugin.json` — bump `version` to the new SemVer.
2. Edit `CHANGELOG.md` — promote the `[Unreleased]` section to `## [X.Y.Z] — YYYY-MM-DD` with today's local date (David's PDT, per CORE convention).
3. Run the build (§6).
4. The build produces `core-claude-cowork-max-plugin-v{X.Y.Z}.zip` + `.SHA256SUMS`.

The build NEVER modifies `plugin.json` or `CHANGELOG.md`. Version bumps are a human decision, not a build automation step.

### Test Builds (Pre-Release)

For local Cowork testing where the version hasn't formally shipped yet, the spec allows the artifact to be produced from the working tree as long as:

- `plugin.json` version is whatever the build is testing (e.g., `1.0.0`).
- The build emits a warning (`[build] WARNING: building test artifact — version 1.0.0 has no companion CHANGELOG entry yet`) if `[Unreleased]` entries exist.
- The zip is treated as a test build and MUST NOT be published to GitHub Releases / SharePoint / marketplace without a clean re-build from a post-promotion tree.

## §6. Build Steps

A conforming build implementation performs these steps in order. Pseudocode is illustrative — actual implementation is a separate artifact.

```
# 1. Resolve version
VERSION = read(.claude-plugin/plugin.json).version
assert valid_semver(VERSION)

# 2. Pre-build validation gates (§10.1)
run_pre_build_validation()  # aborts on failure

# 3. Stage clean copy of source tree
STAGE = mktemp_dir()
rsync source/ STAGE/core-claude-cowork-max-plugin/ \
  with exclusions from §8

# 4. Post-stage validation (§10.2)
run_post_stage_validation(STAGE)  # aborts on failure

# 5. Build zip
OUT = ~/Desktop/core-claude-cowork-max-plugin-v{VERSION}.zip   # local default
zip -r OUT (cd STAGE; zip -rq OUT core-claude-cowork-max-plugin/)

# 6. Post-build validation (§10.3)
run_post_build_validation(OUT)  # aborts on failure

# 7. Emit companion artifacts
shasum -a 256 OUT > OUT.SHA256SUMS
optionally: gpg --armor --detach-sign OUT -> OUT.asc

# 8. Report
print(path, size, sha256, file_count, top-level_entries)
print("READY FOR UPLOAD: Cowork → Customize → Browse plugins → Add custom plugin")
```

Build is idempotent: running twice on the same source tree produces byte-for-byte identical zips (modulo zip timestamps — file content hashes inside are stable).

## §9. README & ARCHITECTURE Content Requirements

Both `README.md` and `ARCHITECTURE.md` MUST exist at the top of the zip, MUST be plugin-specific, and MUST NOT be the corresponding documents from the underlying `core-skill` repo.

### Why this matters

The CORE skill (open source at `dbatesai/core-skill`) has its own `README.md` and `ARCHITECTURE.md` that describe the **skill product** — what `/core` does, the DM persona, swarm orchestration, anti-convergence machinery. Those documents are correct for someone customizing the skill or installing it manually.

This plugin is a different thing: it's the Cowork-native packaging of CORE — hooks, MCP server, Live Artifacts, connector integration, three-channel distribution. End users installing the plugin need plugin-level documentation, not skill-level documentation.

### README.md contract

Must answer for a Cowork end user:
- What is the plugin? (one paragraph)
- Three install channels (marketplace / SharePoint / GitHub releases) with verification steps
- First-session walkthrough (5–7 steps)
- What you get (feature table)
- Which plugin should I install? (decision matrix vs. standard plugin and Claude Code plugin if/when those exist)
- Capability levels (L1/L2/L3) and when each activates
- Troubleshooting top failures
- Cross-platform notes (macOS / Windows / Linux runtime requirements)
- Where to file issues, where to customize the skill (link to `core-skill` repo)

Current `README.md` (2026-05-11) already satisfies this contract.

### ARCHITECTURE.md contract

Must describe for a maintainer or auditor:
- Two-OS architecture: hooks run on host, DM runs in Cowork VM
- Capability ladder (L1 Node MCP / L2 Cowork bridge / L3 chat-only)
- SessionStart flow: probe, capability decision, improvement-queue apply, marker emission
- SessionEnd flow: dm-profile flock, dream-cycle synthesis
- MCP server: 8 read-only tools, file-read scope, no destructive operations
- MCP install flow: probe → write to `claude_desktop_config.json` → restart-required UX
- Live Artifact data flow: DM-broker pattern (DM pulls state DM-side, pushes via `update_artifact`)
- Bundled-skill model: snapshots at release time, CHECKSUMS verification, refresh script
- Trust model: file-tool scope, `request_cowork_directory` per-session approval, supply-chain mitigations
- Threat model summary (TM-1..14 — full version in plugin spec)
- Cross-platform compatibility approach (Python `platform.system()` branching, path semantics)

`ARCHITECTURE.md` does NOT currently exist in this repo. It MUST be authored before the next non-test release. The build will fail (per §10.2) until it exists.

### Forbidden content in plugin docs

- Discussion of `/core-analysis` sub-skill workflows beyond a one-line "the bundled skill includes it" mention (sub-skill docs live in `skills/core/` and the source `core-skill` repo)
- Internal session history, handoffs, or session-log references (those live in the CORE dev project)
- Persona names without context (the DM's name is rendered from `dm-profile.md` at runtime — never hardcoded in plugin docs)

## §10. Validation Gates

Validation runs in three phases. Any gate failure aborts the build and surfaces the specific check that failed.

### §10.1 — Pre-build (source tree)

| Check | Failure mode |
|---|---|
| `plugin.json` parses as JSON | Manifest malformed |
| `plugin.json` has required fields (`name`, `version`, `description`, `author`, `repository`, `license`) | Manifest incomplete |
| `plugin.json` version is valid SemVer | Version malformed |
| `CHANGELOG.md` has a `## [VERSION] — YYYY-MM-DD` header matching `plugin.json` version | Version coherence (§4) — release notes missing |
| `CHANGELOG.md` does not have an `## [Unreleased]` section (unless `--allow-unreleased` flag passed for test builds) | Unreleased work would be silently shipped |
| `ARCHITECTURE.md` exists at repo root | Required plugin doc missing (§9) |
| `README.md` exists at repo root | Required plugin doc missing |
| `LICENSE` exists at repo root | Required file missing |
| `hooks/hooks.json` parses as JSON and references existing scripts under `${CLAUDE_PLUGIN_ROOT}` | Hook manifest broken |
| All referenced hook scripts exist (`scripts/session_start.py`, `scripts/session_end.py` per current hooks.json) | Hook target missing |
| All Python source files in `scripts/` and `mcp-server/` pass `py_compile` | Python syntax error |
| `mcp-server-node/server.mjs` parses with `node --check` | Node syntax error |
| `mcp-server-node/package.json` parses as JSON | Node manifest malformed |
| `skills/CHECKSUMS.txt` matches actual SHA256 of bundled skill files | Skill snapshot drift — re-run `refresh_bundled_skills.py` |
| All six project templates exist under `templates/project-types/` | Required templates missing |
| All four connector docs exist under `connectors/` | Required connector docs missing |
| `live-artifacts/cdn-checksums.txt` pins match the CDN-referenced asset hashes in HTML files | Supply-chain integrity drift |
| SVG security check (no external refs, no embedded scripts) on `live-artifacts/assets/**/*.svg` | Security finding |

### §10.2 — Post-stage (after rsync, before zipping)

| Check | Failure mode |
|---|---|
| All files in §7 Inclusion List exist under staging | Required file lost during staging |
| Zero files in §8 Exclusion List exist under staging (recursive scan) | Excluded content leaked through |
| Exec bits preserved on `scripts/*.sh` companions and any `*.py` that needs to be runnable | Cowork won't be able to invoke hooks |
| Symbolic links resolved or excluded — no symlinks in the staged tree | Cowork unzip doesn't follow links predictably |
| Staging directory size under 45 MB | Headroom check vs. 50 MB Cowork cap |
| Top-level directory in staging is exactly `core-claude-cowork-max-plugin/` (no nested or alternate names) | Zip extraction shape wrong |

### §10.3 — Post-build (zip artifact)

| Check | Failure mode |
|---|---|
| Zip extracts without errors | Corrupt artifact |
| Extracted top-level entry is `core-claude-cowork-max-plugin/` and nothing else | Extraction shape wrong |
| Zip size under 50 MB | Cowork would reject |
| `plugin.json` inside the zip parses as JSON | Manifest corruption during zip |
| Version field inside zipped `plugin.json` matches the version segment in the zip filename | Filename ↔ manifest drift |
| `hooks/hooks.json` inside the zip parses as JSON | Manifest corruption during zip |
| File count in zip matches file count in staging | Files lost during zip |
| SHA256 emitted to `{zipname}.SHA256SUMS` | Companion artifact missing |
| Listed file count, total size, SHA256 reported to caller | Build silent — caller can't verify |

### Human Review Gates (not enforceable by code)

The build cannot verify these. They MUST be reviewed by a human (or by `/core`) before publishing to a public channel:

- `README.md` voice + content is plugin-appropriate, not skill-appropriate (§9 contract)
- `ARCHITECTURE.md` voice + content is plugin-appropriate, not skill-appropriate (§9 contract)
- `CHANGELOG.md` release entry is honest and complete (no silent breaking changes)
- Threat model assumptions in `ARCHITECTURE.md` still hold (no probe-overturned claims)
- No secrets or credentials accidentally committed (manual grep for `key`, `token`, `password`, `secret`)

For local test builds going to David's own Cowork instance, these gates are advisory. For any public release (marketplace / SharePoint / GitHub releases), they're mandatory.

## §11. Failure Modes & Recovery

| Symptom | Likely cause | Fix |
|---|---|---|
| `plugin.json version mismatch` | Forgot to bump version before release build | Edit `plugin.json` → bump version. Re-run build. |
| `CHANGELOG missing entry for X.Y.Z` | Bumped `plugin.json` but didn't promote the `[Unreleased]` section | Edit `CHANGELOG.md` → rename `[Unreleased]` to `[X.Y.Z] — YYYY-MM-DD`. Re-run build. |
| `ARCHITECTURE.md missing` | Doc never authored for the plugin | Write it (§9 contract). Re-run build. |
| `CHECKSUMS.txt drift` | Bundled skill files changed without refresh | Run `python3 scripts/refresh_bundled_skills.py`. Re-run build. |
| `excluded content leaked` | Source tree has files matching §8 patterns that escaped rsync | Add to rsync exclude list; investigate why they're in the tree; clean working copy. |
| `zip over 50 MB` | Accidentally bundled large asset | Inspect `du -sh */` in staging; remove large unintended files. |
| `Cowork rejects upload` | Manifest invalid / top-level structure wrong | Run `unzip -l` on artifact; confirm structure matches §3. |
| `Cowork installs but hooks don't fire` | Exec bit lost during zip | Confirm `unzip -Z artifact.zip 'scripts/*.sh'` shows `rwx` perms. |
| `Cowork installs but MCP doesn't register at SessionStart` | `claude_desktop_config.json` not writable / `CORE_DRY_RUN=1` set | Check `~/.core/mcp-install-log.md` for the error. Re-run install with dry-run cleared. |

## §12. Releasing a New Version — End-to-End Workflow

Authoritative sequence for shipping a new release:

1. **Decide what's shipping.** Review accumulated `[Unreleased]` CHANGELOG entries. Confirm scope is coherent and the version bump (major/minor/patch) matches SemVer rules.
2. **Refresh bundled skills** if any of `core` / `orient` / `finalize` / `vibecheck` changed since last release:
   ```bash
   python3 scripts/refresh_bundled_skills.py
   ```
3. **Update `ARCHITECTURE.md`** if any architectural decision changed (capability ladder, MCP surface, threat model).
4. **Bump version in `plugin.json`** to the new SemVer.
5. **Promote CHANGELOG entries** — rename `[Unreleased]` to `[X.Y.Z] — YYYY-MM-DD` with today's local-day date.
6. **Run dry-run build** to verify validation gates pass:
   ```bash
   ./scripts/build-release.sh --dry-run
   ```
   *(Build script is a future artifact — until then, run validation steps manually.)*
7. **Run actual build:**
   ```bash
   ./scripts/build-release.sh
   ```
   Output lands in `dist/` (or `~/Desktop/` for local test builds).
8. **Locally smoke-test the artifact** — install in your own Cowork, walk through onboarding, verify all capability levels degrade correctly.
9. **For public release only:**
   - Tag the commit (`git tag v{X.Y.Z}`)
   - Push tag (`git push origin v{X.Y.Z}`)
   - Create GitHub Release with the zip + `.SHA256SUMS` + `.asc` attached, populated from `RELEASE_NOTES_TEMPLATE.md`
   - Submit to Anthropic plugin marketplace per their process
   - Upload to SharePoint for org-internal channel (if applicable)
10. **Verify install from at least one published channel** before announcing.

## §14. Downstream Wrapper Distributions

This plugin has a known downstream consumer: **BBLens** — an internal-to-T-Mobile wrapper distribution maintained by the plugin author. BBLens vendors a copy of this plugin's source tree, layers T-Mobile-specific policies and directives on top, and re-packages as a separate plugin (`bblens-plugin`). The sync from this repo into BBLens is **manual**: the maintainer pulls from public GitHub onto a T-Mobile work machine, then copies into the BBLens directory and re-applies the wrapper.

This section exists so the build spec can flag implications for any wrapper distribution — not just BBLens. Anyone forking this plugin to add organization-specific policy on top should read this section first.

### §14.1 — Naming collisions

The plugin name field in `.claude-plugin/plugin.json` is currently `core-plugin`. A wrapper distribution MUST change this to a distinct name (`bblens-plugin`, `<org>-plugin`, etc.) before re-packaging — otherwise both plugins installed on the same machine will collide at manifest level. Specifically, watch for:

- `plugin.json` `name` — primary identifier; conflict will cause one to overwrite the other in plugin storage
- MCP server name registered in `claude_desktop_config.json` — `mcp_server_install.py` writes the entry as `core` (or whatever the script's hardcoded key is); a wrapper MUST change that key or the second install will overwrite the first
- Hook event subscriptions — both plugins firing SessionStart will run both sets of hooks unless the wrapper deduplicates
- Paths written under `~/.core/` — `capability.json`, `mcp-install-log.md`, `dm-profile.md`, `improvement-queue.md` are all SHARED between core-plugin and any wrapper. Wrappers MUST either (a) accept the shared state or (b) scope their writes to a wrapper-specific path (e.g., `~/.bblens/`)
- Skill files installed into `~/.claude/skills/` — bundled skill names (`core`, `orient`, `finalize`, `vibecheck`) will collide with a wrapper that bundles the same names; either prefix (`bblens-core`, etc.) or accept the override

### §14.2 — Version coherence across wrapper layers

A wrapper distribution has its OWN version stream, independent of this plugin's version. The wrapper's `plugin.json` version is the wrapper's version, not this plugin's version. Wrappers should:

- Maintain their own CHANGELOG that documents both wrapper-layer changes AND which upstream `core-plugin` version is vendored
- Pin the upstream version in a known location (e.g., a `UPSTREAM_VERSION` file or a frontmatter field in the wrapper's README) so it's grep-able during audit
- Surface upstream `CHANGELOG.md` entries that affect wrapper behavior — particularly `### Security`, `### Breaking`, or `### Deprecated` headers

This spec's §4 version-coherence rules apply to the WRAPPER'S `plugin.json` ↔ wrapper's `CHANGELOG.md` ↔ wrapper's zip filename, not to the upstream version. Don't confuse the two streams.

### §14.3 — Breaking-change visibility

Because wrapper sync is manual, breaking changes in this plugin propagate slowly. To minimize wrapper-maintainer surprise:

- **CHANGELOG.md entries** in this repo should use `### Breaking`, `### Security`, `### Deprecated` headers explicitly when applicable — wrappers grep on these
- **Manifest shape changes** (added/removed required fields in `plugin.json`, `hooks/hooks.json`, `mcp-server-node/package.json`) deserve a prominent CHANGELOG callout, not buried in `### Changed`
- **Renamed environment variables** (`CORE_*` namespace) deserve the same treatment — wrappers may pass-through, override, or extend env-var semantics
- **Renamed or moved files** in `scripts/`, `hooks/`, `mcp-server-node/`, `skills/` are highest-impact — they break wrapper assumptions about where to layer their additions
- **Hook signature changes** (different stdin JSON shape, different stdout protocol, different env-var contract) affect anything the wrapper layers on top of hooks

When this plugin's spec changes one of the above, the change-author SHOULD note `WRAPPER-IMPACT:` somewhere in the commit message or CHANGELOG entry. This is advisory, not gated.

### §14.4 — License + IP boundary

This plugin is MIT-licensed (see `LICENSE`). Wrapper distributions almost certainly are not — they layer organization-internal policy that is proprietary to that organization. Specific implications:

- Wrapper-specific files (T-Mobile policy text, internal connector docs, internal threat model deltas) MUST NOT be back-ported into this repo
- Pull requests against this repo MUST be reviewed for accidental disclosure of wrapper-internal content
- Public CHANGELOG / release notes MUST NOT name internal wrappers (BBLens is mentioned in this internal build spec, NOT in `README.md` or `CHANGELOG.md`)
- The bundled `LICENSE` file in this repo applies to upstream content only; wrappers ship their own license

### §14.5 — Sync-lag risk

Manual sync introduces a window where upstream has shipped a fix and the wrapper has not yet propagated it. For critical-correctness or security fixes:

- This plugin should ship the fix in a patch release (X.Y.Z+1) with a prominent CHANGELOG `### Security` entry
- The release notes should explicitly tell wrapper maintainers to re-sync ASAP
- For very high-severity fixes, this plugin's maintainer should consider out-of-band notification to known wrapper maintainers (email, Slack, GitHub issue) — not a build automation step, but a manual playbook item

The build spec doesn't enforce this — it's a release-communication discipline. Calling it out here so the discipline isn't invisible.

### §14.6 — What this spec does NOT cover

This section flags wrapper-relevant implications of the upstream build. It does NOT cover:

- The wrapper's own build process — that's the wrapper's spec to write
- Inter-plugin coordination at runtime (when both core-plugin and a wrapper are installed) — that's a runtime concern handled by the plugins themselves, not the build process
- Mechanics of vendoring (git submodule vs. copy vs. release-zip-extract) — wrapper's choice

## §15. Spec Maintenance

This spec is itself a versioned artifact. Changes follow:

- Substantive change (new gate, changed exclusion rule, changed filename convention) → bump spec version in the header line, add an entry to a `## Spec Changelog` section at the bottom.
- Editorial change (clarification, typo, prose polish) → no version bump.
- Any change should be reviewed against the running release — if the next release's build needs to differ from the spec, that's a spec deficiency, not a build override.

If the build script and spec disagree: **the spec is authoritative**. Fix the script.
